간접 참조 최적화를 통한 N+1 문제 해결

문제상황:

매일 새로운 소설이 추가될 때, 축적된 데이터로 인해 사용자들의 전체 소설 조회 시 성능 저하가 우려됨

1,000개의 데이터로 테스트 진행 결과, 기존 JPA를 이용한 방식은 40초의 응답 시간 발생

기존 방식의 문제점:

개별 레코드마다 별도의 쿼리를 실행하여 데이터베이스 부하가 높음

N+1 문제로 인해 데이터 조회 시 불필요하게 많은 쿼리가 발생

해결방법:

1. 간접 참조 대상의 PK를 추출

2. IN 쿼리로 벌크 데이터 조회

3. Map 자료구조 활용으로 O(1) 시간 복잡도의 데이터 접근 구현

4. Map 자료구조 사용에 따른 메모리 사용량 증가 우려 해소를 위해 대용량 데이터 처리 시 청크 처리 적용

결과:

평균 응답 시간이 40초에서 1.8초로 단축

소설 조회 성능 개선으로 사용자 경험 향상

대량의 비동기처리가 들어올 경우 쓰레드는 한정적이기에 부하의 우려가 생김

JWT를 갱신하는 과정에서 클라이언트로부터 동시에 많은 요청이 들어왔을 때 동시성 문제가 발생하는 것을 확인

c이로 인해 사용자가 의도치 않게 로그아웃 처리되는 현상을 발견

기존 방식의 문제점: JWT 갱신 과정에서 동시다발적인 요청 처리 시 경쟁 조건이 발생하여, 토큰 정보가 불일치하거나 덮어씌워지는 문제로 인해 사용자가 예기치 않게 로그아웃되는 현상이 발생

해결방법:

caffeine cache와 MySQL의 네임드 락(Named Lock)을 사용하여 구현

네임드 락으로 요청을 순차적으로 처리하였고, 요청의 결과를 캐싱하여 일정 시간에 들어오는 경신 요청은 캐싱된 값을 반환하도록 함

결과: 네임드 락과 캐싱을 통해 동시성 문제를 해결하여 사용자 세션의 안정성을 크게 향상시켰고, 동시에 시스템의 응답 성능도 개선
